Fazendo analise de logs e eventos em Windows

Novembro 2, 2025 · 10 min · Guilherme Assunção Da Silva

Neste post trago uma simulação básica de análise de logs no Windows, aplicando conceitos que venho estudando na área de Blue Team e resposta a incidentes. O objetivo foi reproduzir, em ambiente controlado, algumas técnicas comuns utilizadas por atacantes e demonstrar como essas ações deixam rastros nos logs do sistema. Nada aqui representa um cenário real ou avançado, mas sim uma introdução prática a como um analista SOC começa a investigar eventos suspeitos.

Antes de iniciar a simulação, habilitei recursos de auditoria que o Windows mantém desativados por padrão. O principal deles foi o Script Block Logging, que faz com que o sistema registre o conteúdo real de scripts PowerShell executados, inclusive comandos enviados de forma ofuscada. Sem essa configuração ativa, boa parte das evidências simplesmente não seria capturada nos logs.

Com o ambiente preparado, executei uma sequência de comandos simulando ações comuns pós-comprometimento: criação de usuário local, adição ao grupo de administradores, criação de tarefa agendada para persistência e execução de um comando encodado em Base64, técnica utilizada para tentar ocultar o que está sendo executado. O ponto de partida da investigação foi uma notificação do Microsoft Defender indicando ameaça detectada. Em um ambiente corporativo real, esse gatilho poderia vir de um SIEM, de um alerta de EDR ou até de uma anomalia identificada durante um monitoramento de rotina.

Com o alerta como ponto de partida, o próximo passo foi buscar eventos potencialmente maliciosos nos logs do sistema. Acessei o Visualizador de Eventos e naveguei até o log Operational do PowerShell, onde o Script Block Logging registra toda atividade de scripts executados. O Event ID 4104 é o principal indicador nesse contexto, pois captura o conteúdo de cada bloco de script, independente de ter sido enviado de forma ofuscada ou não.

Analisando os eventos registrados no mesmo horário do alerta, identifiquei um evento 4104 com a categoria "Executar um Comando Remoto". Ao inspecionar o conteúdo, o Windows já havia decodificado automaticamente o comando que havia sido enviado em Base64, expondo o que foi realmente executado. Isso demonstra na prática o valor do Script Block Logging: a ofuscação não impede o registro, o sistema captura o conteúdo real independente do formato em que chegou.

Com o timestamp dos eventos em mãos, a análise seguiu para o log de Segurança, buscando eventos relacionados a gerenciamento de contas e alterações de privilégios no mesmo período. Os Event IDs 4720 e 4732 registram respectivamente a criação de uma conta de usuário e a adição dessa conta a um grupo privilegiado. Esses dois eventos juntos indicam uma tentativa clássica de criação de backdoor via conta local, garantindo acesso persistente ao sistema mesmo que a sessão original seja encerrada.

Essa simulação deixou claro como ações que parecem simples deixam rastros evidentes nos logs do Windows, desde que a auditoria esteja corretamente configurada. A sequência de eventos, notificação do Defender, comando encodado capturado pelo Script Block Logging e criação de usuário privilegiado, conta a história completa do que aconteceu e em que ordem. Na prática, esse seria o ponto de partida para uma contenção e remediação, removendo o usuário criado, deletando a tarefa agendada e investigando como o acesso inicial foi obtido.

Abaixo estão os comandos executados e que poderia ser executados seguindo a mesma linha em uma invasão real, com ofuscação de comando

# Cria um usuário local chamado porttest, simulando a criação de uma conta backdoor
net user porttest P@ssword123 /add

# Adiciona o usuário ao grupo de Administradores, escalando privilégios
net localgroup Administradores porttest /add

# Cria um segundo usuário para reforçar a persistência
net user porttest2 P@ssword123 /add

# Adiciona o segundo usuário ao grupo de Administradores
net localgroup Administradores porttest2 /add

# Cria uma tarefa agendada que executa PowerShell na inicialização do sistema
# as flags -nop e -w hidden são usadas para tentar ocultar a execução
schtasks /create /tn "WindowsHealthCheck" /tr "powershell.exe -nop -w hidden -c Write-Output test" /sc onlogon /ru System

# Converte um comando para Base64 e executa de forma ofuscada
# técnica usada para tentar esconder o conteúdo real do que está sendo executado
$cmd = "Write-Output 'Simulando aqui um payload encodado'"
$bytes = [System.Text.Encoding]::Unicode.GetBytes($cmd)
$encoded = [Convert]::ToBase64String($bytes)
powershell.exe -EncodedCommand $encoded

# Cria um arquivo EICAR, detectado pelo Defender como ameaça
$eicar = 'X5O!P%@AP[4\PZX54(P^)7CC)7}$EICAR-STANDARD-ANTIVIRUS-TEST-FILE!$H+H*'
Set-Content -Path "$env:TEMP\eicar_test.txt" -Value $eicar

# Busca eventos suspeitos no log de Segurança do Windows
# cobre criação de usuário, adição a grupos, tarefas agendadas e logins
Get-WinEvent -LogName Security | Where-Object {$_.Id -in @(4720,4732,4698,4625,4624,4688)} | Select-Object TimeCreated, Id, Message | Sort-Object TimeCreated -Descending | Format-List

# Busca no Script Block Log os comandos PowerShell executados
# o Event ID 4104 registra o conteúdo real mesmo que tenha chegado ofuscado
Get-WinEvent -LogName "Microsoft-Windows-PowerShell/Operational" | Where-Object {$_.Id -eq 4104} | Select-Object TimeCreated, Message | Sort-Object TimeCreated -Descending | Format-List

# Confirma a detecção do Defender com timestamp e arquivo identificado
Get-MpThreatDetection | Select-Object InitialDetectionTime, ThreatName, Resources | Format-List
